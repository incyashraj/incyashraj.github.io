Front Page Layer36 — if it isn't portable, is it really yours? — Works Correspondence
▸ Lead Story · Singapore Bureau

Founder bets that software ought to travel like a file.

A young technical founder, late of Zurich and now of Nanyang, builds a runtime that lets one binary visit every operating system without changing its passport.

This newspaper is reliably informed that Mr. Yashraj Bhushan Pardeshi — known to readers of the internet as @incyashraj — has been heads-down for some months on a project named Layer36. The notion, in plain English, is this: write your application once, compile it to WebAssembly, and let a small but opinionated runtime carry it across Windows, macOS, Linux, Android, iOS, ChromeOS and the web, asking the host politely for the permissions it needs before doing anything at all.

The runtime is written in Rust and built atop Wasmtime. A working desktop command-line accepts the customary verbs — run, version, doctor — and continuous integration runs the same .wasm across three desktops without flinching. The project is presently at v0.1.0-rc1, accompanied by a threat model, a battery of benchmarks, architectural decision records, and documentation a reader may actually finish.

Mr. Pardeshi, who arrived at this work by way of distributed systems research at the Communication Systems Group of the University of Zurich and an MA in Blockchain at NTU, summarises his ambition with no particular drama: software should be portable like files. The full apparatus — UCap permissions, the Universal API, host adapters — is laid out in subsequent columns.

Reactions, where solicited, have been mild and curious. A separate report on the inside pages enumerates the prior works that, in this paper's judgement, made the present undertaking inevitable.

— Feature, page 2 —

Layer36, in four movements.

Movement I · Source

Your app, compiled once.

Whatever you write — Rust, anything that targets WebAssembly — leaves you in a single .wasm. No platform glue. No six codebases. The binary is the artifact.

The point is not that WebAssembly is fashionable. The point is that a binary that can be re-hosted is a binary that can outlive its original target.

Movement II · Surface

A universal API.

Instead of platform-specific calls, your code speaks one dialect: a common surface for files, processes, network, devices. One vocabulary, many translations downstream.

The Universal API is small on purpose. Less surface means fewer leaks, fewer special cases, and a future in which adding a host is a weekend.

Movement III · Check

UCap, the gatekeeper.

Every call is checked against explicit capability grants. Default-deny. The host knows, in advance, what your app is allowed to touch.

Permissions are a runtime concern, not a marketing one. UCap is what makes "safe" an honest adjective rather than a hopeful one.

Movement IV · Host

Adapters, native & specific.

An approved call leaves the runtime and is translated to native syscalls per platform. Same artifact, anywhere it lands. Windows, macOS, Linux today; the rest, soon.

The long-term goal is plainly stated: one app model across every operating system worth carrying a binary to.
