ucap, briefly — explicit permissions for a default-deny world.

Every syscall is a trust decision. Most runtimes just pretend it isn't.

When a process starts on most operating systems, it inherits authority it never asked for. It can read files it has no business reading. It can open network connections to anywhere. It can spawn subprocesses. All of this is available by default, revocable only through layers of configuration that most developers never touch.

This is sometimes called the "ambient authority" problem. Your program doesn't earn the right to do things — it just has permission because it's running as you. The authority comes from the identity of the process, not from an explicit grant tied to a specific action.

Ambient authority is convenient right up until the moment something goes wrong. Then it's catastrophic.

where i learned this

Building Bouclier made the problem concrete for me. Bouclier is an ERC-7579 permission layer for AI agents on EVM chains — the idea being that an AI agent executing on-chain transactions should only be able to do exactly what it was explicitly permitted to do, and nothing more. Five deployed contracts, 100 Solidity tests, 128,000 fuzz calls, live on Base Sepolia.

The on-chain version of this problem is actually easier to think about than the off-chain version. On a blockchain, every state transition is visible, every permission grant is a signed transaction, and the rules are enforced by the chain itself. You can't have ambient authority because there's no ambient environment — there's only the state you've been given explicit access to.

The challenge I kept bumping into was: why is this so much harder off-chain? Why does the default mode of running software involve handing it keys to everything?

capability discipline, briefly

The object-capability model is an old idea with a clean formulation: a capability is an unforgeable reference that grants the holder the right to perform a specific action on a specific object. You can only do something if you hold a capability for it. You can't guess capabilities, you can't forge them, and you can only pass them to code you explicitly trust with them.

Applied to a runtime, this means: your program has a capability for each resource it's allowed to touch — a specific file, a specific network address, a specific subprocess. Not "files in general." Not "the network." A specific thing, explicitly granted.

This is what UCap implements in Layer36. Every UAPI call — every interaction with the host environment — goes through a capability check. The default is deny. If you haven't been granted the capability, the call fails with a clear error, not a mysterious permission denied from four layers down the stack.

what this looks like in practice

When you run a Layer36 app, the manifest declares what capabilities it needs: fs.read:/path/to/dir, net.connect:api.example.com:443, whatever it actually requires. The user sees this before the app runs. They grant or deny. The runtime enforces it.

A compromised dependency can't exfiltrate your home directory, because it was never granted a capability to read it. A buggy library can't open arbitrary network connections, because it was never granted net.connect:*. The blast radius of any failure is bounded by what you said the app needed.

Default-deny isn't paranoia. It's honesty about what your software actually needs.

the friction is the feature

The common objection is that this is too much friction. Developers won't declare capabilities. Users won't read them. The manifest becomes a wall of checkboxes everyone clicks through.

I think this gets the direction of causality backwards. The reason permission dialogs are walls of checkboxes is that today's permission models are coarse — "access to your files" means the entire filesystem. When permissions are fine-grained and tied to specific resources, the grant is meaningful. "Read files in ~/Documents/invoices" is a sentence a user can evaluate. "Access to your files" is not.

Building Bouclier taught me that when permissions are visible and auditable, people start to actually care about them. The on-chain permission model worked because every grant was a transaction you could inspect. UCap is an attempt to bring that quality to software that runs off-chain.
