why software should be portable like files.

A PDF just works everywhere. An .mp3 plays on everything. Why is running an app still a platform negotiation?

Here is a thing nobody finds strange: you send someone a PDF and it opens. You don't ask what operating system they're on. You don't ship a Mac version and a Windows version of your PDF. You don't maintain a "PDF runtime" for each platform. The file just works, because the format was designed around portability from the start.

Now try shipping an application. Suddenly you have six codebases — or one codebase and a build matrix that takes four hours to green. Every platform has its own syscall dialect, its own permission model, its own idea of what a "process" is allowed to do. The same logic, rewritten six times, rotting at six different rates.

Software got stuck in a world where portability means "rewrite it for each host." Files never accepted that deal.

where this came from

I spent six months at the Communication Systems Research Group at UZH studying distributed systems — how things coordinate across boundaries they don't control. Then I built Karana OS, an intent-driven OS for IoT devices: custom kernel, scheduler, hardware abstraction layer, 200,000 lines. The whole project was a lesson in what happens when you don't think carefully about the boundary between your logic and the platform underneath it. You end up with code that is deeply married to one host, impossible to move.

Then I built Bouclier — an ERC-7579 permission layer for AI agents on EVM chains. Five deployed contracts, 128,000 fuzz calls, a full threat model. The core insight there was about capability discipline: the idea that every action an agent takes should be explicitly permitted, and that permission should be checked at a well-defined boundary, not assumed.

Those two projects pointed at the same problem from different directions. One was about portability. The other was about permission. Layer36 is what happens when you take both seriously at the same time.

the actual idea

WebAssembly gives you a portable execution format. A .wasm binary is deterministic, sandboxed, and runs the same on any conforming runtime. That's the first ingredient.

The second ingredient is a small, opinionated API surface — what I call UAPI. Instead of letting your app talk directly to the OS (files, network, processes, all with different syscall conventions on each platform), it talks to UAPI. One dialect. The runtime translates that into whatever the host OS actually speaks underneath.

The third ingredient is capability checks by default. Every UAPI call goes through UCap — a permission layer that defaults to deny. Your app declares what it needs. The user grants it. Nothing else runs. No ambient authority, no implicit permissions inherited from the process that launched you.

Write once. Capability-checked. Ride the host adapter wherever you land.

why default-deny matters

Most software runs with more authority than it needs. A notes app that can read your entire filesystem. A CLI tool that can make arbitrary network requests. We accept this because the alternative — explicitly granting permissions — felt like too much friction.

But the friction is the point. Capability discipline forces you to be honest about what your software actually needs. It makes the permission surface visible. It makes it auditable. And it means a compromised dependency can only do what you said it could do — not everything the OS will let a process do.

This isn't a new idea. It goes back to the object-capability model, to Plan 9, to a long tradition of people who thought carefully about the boundary between a program and its environment. Layer36 is an attempt to make that tradition practical for the kind of software people ship today.

where we are

Layer36 v0.1.0-rc1 ships a desktop runtime in Rust with Wasmtime, a CLI with run, version, and doctor, UAPI for CLI-style apps, UCap capability checks, and the same .wasm binary running on Windows, macOS, and Linux. There's a full threat model, architecture decision records, and public docs.

It's early. The surface is deliberately small. But the same binary runs on three platforms today — and when we extend the host adapters, it will run on more without the app changing at all.

That's the bet. Software that travels like files.
