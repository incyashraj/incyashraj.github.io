universal protocols are becoming important in the ai era.

MCP connects agents to tools. A2A connects agents to other agents. But where does the generated software actually run — and who checks what it's allowed to do?

A few months ago, most AI systems were isolated. You gave a model a prompt, it gave you text. That was the whole interaction. Now those same models are being wired into tools, databases, calendars, file systems — and starting to talk to each other. The AI stack is getting connected, and to do that it needs shared languages.

That's what these protocols are about. Not hype, not marketing — actual interfaces that let different parts of an AI system understand each other without custom glue code.

mcp: a shared plug standard

Before MCP, if you wanted your AI assistant to read a file, call an API, or search a database, someone had to write a custom integration for every pairing. Every model, every tool, its own connector. It was messy and it didn't scale.

The Model Context Protocol solves this the same way USB solved the connector problem. You define a server that exposes a capability — a file reader, a web search, a calendar, whatever — and any MCP-compatible client can talk to it. The protocol is the common language. You build a tool once and every MCP client can use it.

MCP gives AI agents hands. They can reach out and actually touch things.

Anthropic published MCP as an open standard, and it spread quickly. OpenAI, Google, and a growing number of developer tools now support it. When a protocol catches on this fast, it usually means it was solving a real pain.

a2a: agents talking to agents

MCP handles the agent-to-tool relationship. A2A — Agent to Agent — handles what happens when agents need to talk to each other.

As AI systems get more capable, people are building pipelines where one agent orchestrates others. A planning agent delegates to a research agent, which delegates to a writing agent. But if those agents were built by different teams, on different frameworks, they can't coordinate without someone writing the translation layer.

A2A is Google's answer to that. An orchestrator agent can hand off a task to a specialized agent and get structured results back — without caring about how that agent was built underneath. They share a protocol, so they can collaborate.

the question neither of them answers

MCP and A2A cover a lot of ground. But there's a question they leave open: once an agent creates a small tool or app, where does that actually run? And how does the machine know it's safe to let it run?

Today, AI coding tools can write a script, generate a CLI utility, or produce a working app. But the output is usually a loose file — a Python script, a shell command, a compiled binary. You have to decide whether to run it. If you do run it, it probably has access to your entire file system, can make arbitrary network requests, and can read things it has no business reading. That's just how most software works — it inherits whatever permissions the operating system is willing to give a process.

When the author is a person you trust, that's usually fine. When the author is an automated system that generated a hundred small tools this week, it's a harder question.

The problem isn't whether AI can write code. It's whether that code can run somewhere safe.

where layer36 fits in

This is the gap I built Layer36 to sit in. Not between agent and tool — that's MCP. Not between agent and agent — that's A2A. Between generated software and the host machine.

The idea has three parts. First, WebAssembly as the execution format — a .wasm binary is portable and sandboxed by design. The same file runs on macOS, Windows, and Linux without any changes. Second, a single API surface called UAPI — instead of letting apps talk directly to the OS using each platform's different syscall conventions, everything goes through one interface. The runtime translates it for the host underneath.

Third, and most important, capability checks by default. Every UAPI call goes through UCap, a permission layer that defaults to deny. An app declares what it needs — read this directory, open a network connection, write output. The user grants it. Nothing else runs. No silent access, no ambient authority.

why default-deny matters

Most software runs with more authority than it actually needs. A notes app that can read your whole file system. A CLI tool that can make any network request. We accept this because asking users to grant permissions for every little thing felt like too much friction.

But capability discipline forces you to be honest. You have to say what your app actually needs. That permission surface is then visible — you can read it in a manifest, audit it, restrict it. And if something goes wrong, the damage is bounded. A compromised dependency can only do what you said it could do, not everything the OS will allow.

This isn't a new idea. It goes back to the object-capability model, to Plan 9, to decades of thinking about the right boundary between a program and its environment. Layer36 is an attempt to make that practical for software people actually ship today.

how the three fit together

MCP, A2A, and Layer36 each sit at a different boundary in the same stack. An agent might use MCP to pull context from a database, coordinate with a specialist agent over A2A, write a small utility in response, and ship that utility as a Layer36 artifact that runs safely on your machine.

  • MCP: agent ↔ tools and data
  • A2A: agent ↔ agent
  • Layer36: generated software ↔ host machine

None of them replace each other. They're solving different problems at different layers. The pattern they share is the same one: define a clear boundary, give it a protocol, and let the ecosystem build around it.

where this is going

Layer36 v0.1.0-rc1 is out now — a desktop runtime in Rust with Wasmtime, a CLI, UAPI for CLI-style apps, UCap capability checks, and the same .wasm binary running on Windows, macOS, and Linux. It's early, the surface is deliberately small, but the portable execution is working.

The longer path is extending the host adapters — more capabilities, more platform surface, eventually mobile feasibility. The goal stays simple: software that travels like a file, runs in a known sandbox, and asks before it touches anything.

If AI is going to generate more software, the runtime is where trust actually gets enforced.
