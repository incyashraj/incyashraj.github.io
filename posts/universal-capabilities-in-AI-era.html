universal protocols are becoming important in the ai era.

MCP connects agents to tools. A2A connects agents to other agents. The next missing boundary is execution.

The AI stack is moving from isolated chatbots toward connected systems. Agents will call tools, talk to other agents, create small apps, and generate local utilities.

When that happens, the hard problem is no longer only whether code can be written. It is whether the software can be run safely, moved across machines, and trusted by the user.

If AI creates more software, the runtime becomes the trust boundary.

That is why universal protocols matter. They reduce custom integration work. They make ecosystems easier to connect. They create shared boundaries that developers can build around.

This is good for Layer36 because Layer36 is built around the same direction, but at a different layer of the stack.

the ai stack is standardizing

The clearest example is the Model Context Protocol, or MCP. MCP gives AI applications a common way to connect with tools, APIs, files, databases, and external systems. Instead of every model and every tool needing a separate integration, the ecosystem can converge around a shared interface.

Agent2Agent, or A2A, points in another direction. It gives agents a way to communicate and collaborate with other agents, even when they are built by different teams or frameworks.

These protocols are not the whole AI stack. They are signs of where the stack is going. AI systems need common language between parts that used to be isolated.

the missing boundary is execution

MCP can help an agent call a tool. A2A can help agents talk to one another. But once an agent or developer creates a local tool, there is still a practical question:

Where does that software run, what can it access, and how does the user know it is safe?

Today, this is still fragmented. Generated scripts often become loose files. Small utilities become platform-specific builds. Internal tools are copied around without a clear permission model. The user is left to trust the source, inspect the code, or run it anyway.

That is not enough if AI dramatically increases the number of small software artifacts.

why this creates value for layer36

Protocols become valuable when they sit at a recurring boundary. For Layer36, that boundary is the moment software tries to become real on a machine.

Layer36 is not only trying to make apps cross-platform. The deeper bet is that portable software also needs a permission model. An app should not silently reach into the file system, network, clipboard, microphone, or local storage just because it can run.

The value comes from combining three things:

  • Portability. One app artifact should be able to move across hosts.
  • Permission clarity. The host should know what the app is asking for before it grants power.
  • Native behavior. Approved requests should still become real behavior on the machine through host adapters.

If Layer36 gets this right, it becomes a control point for AI-era software. Developers can build small tools faster. Users can run them with more confidence. Companies can define what generated software is allowed to do.

why now

The timing matters. AI coding tools are making software creation cheaper. At the same time, the broader AI infrastructure stack is moving toward interoperability. MCP and A2A are not just technical details. They show that the market is becoming comfortable with protocol-shaped infrastructure.

Layer36 takes that same movement and applies it to runtime execution.

The claim is not that Layer36 replaces MCP or A2A. It lives at a different boundary.

MCP is about reaching tools and context. A2A is about agent collaboration. Layer36 is about running software safely and portably on the host.

the investor lens

From an investor perspective, this matters because protocol layers can become infrastructure. They are not only features. They can become the place where developers build, users trust, and ecosystems form.

Layer36 is still early. The current proof is a runtime foundation: WebAssembly components, a universal API surface, capability checks, manifests, sample apps, and Phase 3 work toward a desktop developer preview. The next step is developer proof.

The question is not whether the world wants another framework. The better question is:

In a world where agents generate more software, who owns the safe execution boundary?

Layer36 is my answer to that question.

where this goes

If the thesis is right, Layer36 can grow from a runtime proof into a developer platform. First, safe portable local tools. Then a desktop developer preview. Then SDKs, packaging, mobile feasibility, distribution foundations, and eventually a trusted app layer for AI-generated software.

The long-term outcome is simple to say and hard to build: software should be easier to move, safer to run, and easier to trust.
